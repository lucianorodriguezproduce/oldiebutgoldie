# ADR-001: Firebase como Núcleo de Tiempo Real

## Estado
Aceptado

## Contexto
El búnker requiere una sincronización instantánea entre el Admin y los Usuarios para negociaciones (`trades`), actualización de stock (`inventory`) y configuración global (`site_config`). Las bases de datos SQL tradicionales requerirían capas complejas de WebSockets o polling costoso.

## Decisión
Utilizar **Firebase (Firestore + Auth)** como motor soberano:
- **Firestore**: Para persistencia NoSQL con listeners nativos (`onSnapshot`).
- **Firebase Auth**: Para gestión de identidad sin fricciones (Google, Email/Root).

## Consecuencias
- **Positivo**: Latencia casi nula en actualizaciones de UI. Escalabilidad horizontal automática.
- **Negativo**: Dependencia del ecosistema Google Cloud. Consultas complejas (joins) deben ser resueltas en la capa de servicio (como `archivoService`).
